Чтобы проанализировать coredump и найти проблемный запрос, выполните следующие шаги:

### 1. **Найдите путь к coredump и бинарнику suhariki**

- Обычно coredump сохраняется в `/var/lib/systemd/coredump/` или текущей директории. Проверьте:
    ```bash
    coredumpctl list | grep suhariki
    ```
- Если coredump найден, экспортируйте его:
    
```bash
coredumpctl -o /tmp/core dump suhariki
```
    
- Убедитесь, что бинарник `suhariki` доступен (например, `/usr/bin/suhariki`).

### 2. **Запустите GDB**
```bash
gdb /usr/bin/suhariki /tmp/core
```
### 3. **Анализ стека вызовов**

Внутри GDB выполните:

```gdb
(gdb) bt
```

**Пример вывода:**

```gdb
#0  0x00007f3e4a1a5d21 in __GI___strlen_avx2 () from /lib/x86_64-linux-gnu/libc.so.6
#1  0x000055d6e3b0225f in handle_request (request=0x0) at server.c:42
#2  0x000055d6e3b0278a in main (argc=1, argv=0x7ffd8e1d3b28) at server.c:101
```

**Интерпретация:**  
Программа упала в `strlen`, вызванной из `handle_request`, куда передан нулевой указатель (`request=0x0`).

---

### 4. **Проверьте значения переменных**

Перейдите в фрейм, где произошла ошибка:

```gdb
(gdb) frame 1
```

Выведите аргументы и локальные переменные:

```gdb
(gdb) info args
(gdb) info locals
```

**Пример вывода:**

```gdb
request = 0x0
```

---

### 5. **Исследуйте память**

Найдите данные запроса, если они есть в памяти. Например, поищите строку:

```gdb
(gdb) x/s 0x7ffd8e1d3a00
```

**Пример вывода:**

0x7ffd8e1d3a00: "GET /api?data=AAAAAAAAAAAAAAAAAAAA... HTTP/1.1"

Здесь видно, что передан очень длинный параметр `data`.

---

### 6. **Определите причину падения**

В данном примере:

- `handle_request` получила `request=0x0`, что привело к попытке вычислить длину нулевого указателя через `strlen`.
    
- Это указывает на то, что запрос не был корректно инициализирован (например, клиент отправил пустой или битый HTTP-запрос).
